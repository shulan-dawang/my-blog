面向对象设计的基本原则

2026 年 5 月 13 日

前面我们学习了类和对象、继承与接口，这些都是面向对象编程的工具。 但有了工具还不够——你还需要知道怎么用才漂亮。 这篇文章聊聊面向对象设计的五个核心原则，它们是写出可维护、可扩展代码的底层心法。

UML 类图简介

在设计阶段，与其直接写代码，不如先画图。UML（Unified Modeling Language，统一建模语言）的类图 是最常用的设计表达工具。一张图胜过一百行注释。

一个标准的 UML 类图通常包含三格：

  • 第一格：类名（抽象类用斜体，接口用 «interface» 标注）
  • 第二格：属性（格式：可见性 属性名: 类型）
  • 第三格：方法（格式：可见性 方法名(参数): 返回类型）

类与类之间的关系用不同的线条表示：

  • 实线 + 空心三角：继承（extends），表示 "is-a" 关系。
  • 虚线 + 空心三角：实现接口（implements），表示 "can-do" 关系。
  • 实线 + 普通箭头：关联（Association），一个类持有另一个类的引用。
  • 实线 + 空心菱形：聚合（Aggregation），整体与部分的关系，部分可以独立存在（如汽车与轮胎）。
  • 实线 + 实心菱形：组合（Composition），更强的整体-部分关系，部分不能独立存在（如人和心脏）。
  • 虚线 + 普通箭头：依赖（Dependency），一个类的方法中临时使用了另一个类。

不用背，多看多画自然就记住了。关键是理解每种关系表达的耦合程度从强到弱： 继承 > 实现 > 组合 > 聚合 > 关联 > 依赖。

面向抽象原则

面向抽象编程（Program to an Interface, not an Implementation）是面向对象设计的第一原则。 它的核心思想是：代码应该依赖抽象（接口或抽象类），而不是具体的实现类。

先看一个反例——依赖具体类：

public class Driver {
    // 依赖具体的 Benz 类
    public void drive(Benz benz) {
        benz.start();
        benz.run();
    }
}

问题很明显：如果换成 BMW 或者 Audi，Driver 类就得改。 违背了"对扩展开放，对修改关闭"的原则（后面会讲）。

改成面向接口的写法：

// 定义抽象
public interface Car {
    void start();
    void run();
}

// 具体实现
public class Benz implements Car {
    @Override
    public void start() { System.out.println("奔驰启动"); }
    @Override
    public void run()   { System.out.println("奔驰行驶"); }
}

public class BMW implements Car {
    @Override
    public void start() { System.out.println("宝马启动"); }
    @Override
    public void run()   { System.out.println("宝马行驶"); }
}

// Driver 只依赖接口，不依赖具体品牌
public class Driver {
    public void drive(Car car) {   // 参数是接口类型
        car.start();
        car.run();
    }
}

现在 Driver 可以开任何实现了 Car 接口的车—— 新增奥迪、特斯拉都不用改 Driver 的代码。

这条原则在 Java 标准库中随处可见：比如 List<String> list = new ArrayList<>()， 左边写接口类型，右边写具体实现。将来换成 LinkedList，调用方代码一行不动。

"开-闭"原则（Open-Closed Principle）

"开-闭"原则是 Bertrand Meyer 在 1988 年提出的经典设计原则，也是 SOLID 中的 O。 一句话概括：对扩展开放，对修改关闭。

什么意思？当需求变化时，应该通过新增代码来扩展功能， 而不是修改已有的、已经测试通过的代码。修改老代码风险大——可能引入新 bug，牵一发而动全身。

一个违反开闭原则的案例

假设有个图形绘制系统：

public class GraphicEditor {
    public void draw(String type) {
        if (type.equals("circle")) {
            System.out.println("画圆形");
        } else if (type.equals("rectangle")) {
            System.out.println("画矩形");
        }
        // 每加一种图形，就要改这个方法！
    }
}

每次新增图形类型，都要修改 draw 方法——这是典型的"对修改开放"。

用多态实现开闭原则

重构后：

public interface Shape {
    void draw();
}

public class Circle implements Shape {
    @Override
    public void draw() { System.out.println("画圆形"); }
}

public class Rectangle implements Shape {
    @Override
    public void draw() { System.out.println("画矩形"); }
}

public class GraphicEditor {
    public void draw(Shape shape) {  // 依赖抽象
        shape.draw();                // 多态调用
    }
}

现在要加三角形，只需新建一个 Triangle 类实现 Shape 接口—— 不用动 GraphicEditor 一行代码。这就是对扩展开放，对修改关闭。

总结：面向抽象原则是指导思想，开闭原则是衡量标准。 判断一段代码好不好，就问自己："加新功能时要不要改老代码？"

"多用组合，少用继承"原则

这个原则出自《Design Patterns》（GoF），原文是： "Favor composition over inheritance"（多用组合，少用继承）。 这是面向对象设计中最容易被误解的一条——继承不是不能用，而是不能滥用。

继承的问题

继承看似方便（子类自动获得父类所有代码），但它有三大隐患：

  • 破坏封装：子类依赖父类的内部实现细节，父类一改，子类可能全部崩溃。
  • 灵活性差：继承是静态关系，编译时绑定，运行时没法改变。狗一辈子是动物。
  • 层次膨胀：比如要一个"会飞的狗"和一个"会飞的猫"，用继承很难设计，容易搞出臃肿的类层次。

组合的优势

组合（Composition）是指一个类持有另一个类的引用，通过委托来复用代码： 

// 用接口定义能力
public interface FlyBehavior {
    void fly();
}

public class FlyWithWings implements FlyBehavior {
    @Override
    public void fly() { System.out.println("扇动翅膀飞行"); }
}

public class CannotFly implements FlyBehavior {
    @Override
    public void fly() { System.out.println("不能飞"); }
}

// 通过组合赋予能力
public class Dog {
    private FlyBehavior flyBehavior;  // 组合——持有引用

    public Dog(FlyBehavior fb) {
        this.flyBehavior = fb;
    }

    public void performFly() {
        flyBehavior.fly();            // 委托
    }

    public void setFlyBehavior(FlyBehavior fb) {  // 运行时也能换！
        this.flyBehavior = fb;
    }
}

使用：

Dog normalDog = new Dog(new CannotFly());
normalDog.performFly();  // 不能飞

Dog superDog = new Dog(new FlyWithWings());
superDog.performFly();   // 扇动翅膀飞行

// 运行时动态改变行为
superDog.setFlyBehavior(new CannotFly());  // 超狗退役了
superDog.performFly();   // 不能飞

组合的核心优势：运行时灵活切换、每个组件独立测试、不会形成臃肿的类层次。 这也是策略模式（Strategy Pattern）的基础。

什么时候用继承，什么时候用组合？

判断标准很简单：

  • 用继承：当关系是真正的 "is-a"（狗是动物），且父类行为稳定，不会频繁变动。
  • 用组合：当关系是 "has-a"（狗有飞行能力），或行为需要灵活替换、运行时切换。

如果拿不准——先用组合。组合的代价远小于滥用继承带来的维护灾难。

"高内聚-低耦合"原则

高内聚、低耦合（High Cohesion, Low Coupling）是衡量软件设计质量的两个核心指标。 它不是一个具体的编码技巧，而是一把设计尺子——量一量你的代码，看看哪里需要改进。

内聚（Cohesion）

内聚衡量的是一个类内部各个成员之间的关联程度。 高内聚意味着类的每个属性和方法都与这个类的职责紧密相关——各司其职，不干分外的事。

低内聚的反例——一个类啥都干：

public class GodClass {
    // 数据库连接
    public void connectDB() { ... }
    // 发邮件
    public void sendEmail() { ... }
    // 生成报表
    public void generateReport() { ... }
    // 播放音乐
    public void playMusic() { ... }
}

这个类有四个完全不相关的职责——内聚极低，难以理解、难以测试、难以复用。

高内聚的写法：拆成四个职责单一的类——DatabaseConnector、EmailSender、ReportGenerator、MusicPlayer。

耦合（Coupling）

耦合衡量的是类与类之间的依赖程度。 低耦合意味着一个类的改动不会像多米诺骨牌一样波及到其他类。

高耦合的反例：

public class OrderService {
    private MySQLOrderDao dao = new MySQLOrderDao();  // 强依赖具体类
    private AlipayService pay  = new AlipayService();  // 强依赖具体类

    public void checkout() {
        dao.save();    // 直接调具体实现
        pay.pay();     // 直接调具体实现
    }
}

换成低耦合的写法——依赖接口：

public class OrderService {
    private OrderDao dao;        // 依赖接口
    private PayService pay;      // 依赖接口

    public OrderService(OrderDao dao, PayService pay) {
        this.dao = dao;
        this.pay = pay;
    }

    public void checkout() {
        dao.save();
        pay.pay();
    }
}

将来数据库从 MySQL 换成 PostgreSQL，支付从支付宝换成微信—— OrderService 代码不动，只需传入不同的实现。这就是低耦合带来的威力。

内聚与耦合的关系

两者往往是此消彼长的关系：高内聚的类通常职责单一、依赖简单，自然就是低耦合； 低内聚的"万能类"往往跟谁都有关联，耦合自然就高。所以追求高内聚，往往会顺带降低耦合。

常见的耦合来源

  • 直接 new 具体类：改用工厂模式或依赖注入。
  • 调用静态方法：静态方法绑定死实现，难以替换。
  • 暴露内部数据结构：比如直接返回 List 的内部引用，调用方可能修改到你的数据。
  • 链式调用：a.getB().getC().doSomething() 违反了迪米特法则（最少知识原则）。

小结

今天我们学习了面向对象设计的五个基本原则，它们不是孤立的，而是一个整体：

  • UML 类图：设计的通用语言，用简单的图形表达类结构和关系。
  • 面向抽象原则：代码依赖接口而非具体实现——这是所有原则的根基。
  • "开-闭"原则：对扩展开放，对修改关闭——加新功能不动老代码。
  • "多用组合，少用继承"：has-a 优先于 is-a，运行时灵活、耦合更低。
  • "高内聚-低耦合"：类内部拧成一股绳，类之间尽量松绑。

这些原则说到底就是一句话：让代码更容易改、更容易测、更容易复用。 不必死记硬背，写代码时多回头看看，慢慢就能内化成自己的设计直觉。 下一课我们将用这些原则，实际重构一个案例项目。
