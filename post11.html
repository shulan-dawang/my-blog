使用 Thread 子类创建线程 VS 使用 Runnable 接口创建线程的区别

2026 年 6 月 26 日

在 Java 中，创建线程有两种基本方式：继承 Thread 类并重写 run() 方法， 或者实现 Runnable 接口并将其传递给 Thread 的构造器。两者看起来都能"开一个新线程跑一段代码"， 但它们的设计意图、灵活性和实际适用场景有本质区别。本文将对这两种方式做一次系统的对比。

方式一：继承 Thread 类

最基本的用法——定义一个类继承 Thread，重写 run() 方法，然后调用 start()：

// 方式一：继承 Thread 类
class MyThread extends Thread {
    @Override
    public void run() {
        System.out.println("线程 " + getName() + " 正在执行");
        // 线程要执行的任务
    }
}

// 使用
MyThread t1 = new MyThread();
t1.start();  // 启动新线程，JVM 会调用 run()

这种方式直观且代码量少——线程和任务"融为一体"，一个线程就是一个独立的执行单元。

继承 Thread 的优点

  • 编写简单直接——创建一个子类、重写一个方法即可，适合快速原型。
  • 可以直接操作线程的 API——在 run() 内部，可以直接调用 getName()、getId()、interrupt() 等 Thread 的方法，不需要额外获取当前线程引用。
  • 每个实例天然对应一个线程——对象和线程的身份是统一的，在某些需要为线程附加状态的场景中比较自然。

继承 Thread 的缺点

  • Java 的单继承限制——一个类只能继承一个父类。如果你的类已经继承了其他类（例如 extends Animal），就无法再继承 Thread。
  • 任务和线程机制耦合——线程本身是"执行任务的载体"，而任务是"要执行的内容"。将两者绑定在同一个类中违反了关注点分离（Separation of Concerns）原则。
  • 不易共享任务——你无法将同一个 MyThread 的任务逻辑交给线程池去管理（因为每个 MyThread 的实例只能启动一次）。
  • 资源开销较大——每个线程对象都继承了 Thread 类的完整"重量级"结构，创建大量线程时开销显著。

方式二：实现 Runnable 接口

定义一个类实现 Runnable 接口，实现 run() 方法，然后将其实例传给 Thread 的构造器：

// 方式二：实现 Runnable 接口
class MyTask implements Runnable {
    @Override
    public void run() {
        System.out.println("线程 " + Thread.currentThread().getName() + " 正在执行");
        // 要执行的任务
    }
}

// 使用
MyTask task = new MyTask();
Thread t1 = new Thread(task);
t1.start();

注意一个细节：在 Runnable 的 run() 里访问当前线程需要通过 Thread.currentThread() 静态方法，而不是像继承 Thread 时那样直接调用。

实现 Runnable 的优点

  • 不受单继承限制——你的类可以自由地继承其他父类，同时实现多个接口。
  • 任务与线程分离——Runnable 只描述"要做什么"，Thread 负责"怎么执行"。这种分离使得代码更清晰、更符合面向对象设计原则。
  • 任务可被复用——同一个 Runnable 实例可以交给多个 Thread，也可以提交给线程池（ExecutorService），灵活度远高于继承 Thread。
  • 资源更轻量——Runnable 只是一个接口，任务对象不包含 Thread 的额外开销。
  • 适合线程池——Java 并发包（java.util.concurrent）中的线程池、FutureTask、CompletableFuture 等都接受 Runnable/Callable，不能直接使用 Thread 子类。

实现 Runnable 的缺点

  • 不能直接使用 Thread 的 API——必须通过 Thread.currentThread() 获取当前线程引用后才能调用线程方法，稍显繁琐。
  • 不能直接抛出受检异常——Runnable.run() 的签名没有 throws 子句，受检异常必须在内部 try-catch 处理。
  • 没有返回值——Runnable 的 run() 返回 void，如果需要返回值，需要使用 Callable<V>（它实质上是 Runnable 的增强版）。

核心对比一览

比较维度 继承 Thread 实现 Runnable
继承关系 占用唯一的单继承名额 不占用，可自由继承其他类
设计模式 任务与线程耦合 任务与线程分离（关注点分离）
任务复用 一个线程实例只能启动一次 同一任务可被多个线程/线程池复用
访问线程 API 直接使用 this.getName() 等 需通过 Thread.currentThread()
线程池兼容 不兼容，需额外包装 原生兼容 ExecutorService
资源开销 较重（携带 Thread 全部结构） 较轻（仅一个接口实现）
适用场景 简单场景、教学演示 生产环境、线程池、复杂并发

深入分析：为什么 Runnable 是更好的设计

1. 从面向对象设计角度看

继承 Thread 意味着你的类"是一个线程"（is-a Thread），但大多数情况下，你的业务对象本身"不是线程"—— 它只是"有一些任务需要在线程中执行"。用继承来表达"我要开个线程跑一段逻辑"是被迫使用继承机制来实现组合逻辑， 这本质上是继承的误用。

实现 Runnable 则意味着你的类"是一个可执行的任务"（is-a Runnable），这更贴近真实语义。 然后你把任务交给线程去跑——这是典型的组合优于继承（Composition over Inheritance）的实践。

2. 从并发编程演进角度看

Java 从 1.0 开始就提供了 Thread 和 Runnable，但 Java 5 引入 java.util.concurrent 包后，直接操作 Thread 已经被视为一种初级做法。 现代 Java 并发编程的标准范式是： 

// 现代 Java 并发编程的推荐写法
ExecutorService executor = Executors.newFixedThreadPool(4);

// 提交 Runnable 任务
executor.submit(() -> {
    System.out.println("任务在线程池中执行");
});

// 提交 Callable 任务，可以获取返回值
Future<String> future = executor.submit(() -> {
    Thread.sleep(1000);
    return "任务完成";
});

String result = future.get();  // 阻塞等待结果
executor.shutdown();

在这个范式中，Thread 只作为底层机制存在，我们操作的对象是 Runnable/Callable 和线程池，而不是裸线程。继承 Thread 的方式无法融入这套体系。

3. 从源码角度看——Thread 本身就实现了 Runnable

翻开 JDK 源码，你会发现一个关键事实：

public class Thread implements Runnable {
    // ...
    private Runnable target;  // 构造器中传入的 Runnable

    @Override
    public void run() {
        if (target != null) {
            target.run();  // 如果传入了 Runnable，就调用它的 run()
        }
        // 如果子类重写了 run()，则执行子类的版本（多态）
    }
}

这就揭示了两种方式的本质区别：

  • 继承 Thread 并重写 run()：你替换了 Thread 自身的 run() 实现（多态覆盖）。
  • 实现 Runnable 并传给 Thread：你提供了一个 target，Thread 的 run() 在内部委托给这个 target（策略模式）。

Thread 的源码设计者已经为你选择了策略模式——把"要执行的任务"作为参数注入。 继承 Thread 反而是绕过了这个设计，直接用多态覆盖了策略分发逻辑。

实际编程中的简化写法

在实际开发中，很少有人为每个线程任务定义一个独立的类。更常见的做法是使用匿名内部类或 Lambda 表达式：

匿名内部类方式（Java 7 及之前）

// 使用 Runnable 的匿名内部类
new Thread(new Runnable() {
    @Override
    public void run() {
        System.out.println("匿名内部类线程");
    }
}).start();

Lambda 表达式方式（Java 8+，推荐）

// Runnable 是函数式接口，可以用 Lambda
new Thread(() -> {
    System.out.println("Lambda 线程");
}).start();

// 或者更简洁的单行版本
new Thread(() -> System.out.println("Hello from thread")).start();

这也是 Runnable 方式的一个隐性优势——因为 Runnable 是函数式接口 （只有一个抽象方法的接口），所以可以直接用 Lambda 表达式，代码简洁度远超继承 Thread。

需要注意的共同问题：资源共享与线程安全

无论使用哪种方式创建线程，多线程环境下的资源共享问题都是一样的。如果一个 Runnable 实例被多个线程共享，它的成员变量就是这些线程的共享数据： 

class Counter implements Runnable {
    private int count = 0;  // 多个线程共享这个变量！

    @Override
    public void run() {
        for (int i = 0; i < 1000; i++) {
            count++;  // 非原子操作，会出现竞态条件
        }
    }
}

Counter counter = new Counter();
Thread t1 = new Thread(counter);
Thread t2 = new Thread(counter);  // 两个线程共享同一个 counter
t1.start();
t2.start();

t1.join();
t2.join();
System.out.println(counter.getCount());  // 很可能不是 2000！

解决方法：使用 synchronized、AtomicInteger 或其他并发工具类来保证线程安全。 但这不是本文的重点——这里的重点是继承 Thread 同样存在此问题，两种方式在"线程安全性"上没有任何区别。

小结

继承 Thread 和实现 Runnable 都能创建线程，但在工程实践中，实现 Runnable 是更推荐的方式，原因归纳如下：

  • 设计层面：Runnable 将任务和线程解耦，遵循"组合优于继承"原则，并利用了策略模式。
  • 语言层面：Java 单继承意味着继承 Thread 会浪费唯一的继承名额。
  • 复用层面：Runnable 可以被多个线程共享、提交给线程池、包装为 FutureTask 等。
  • 演进层面：Java 5 之后的并发 API 全部围绕 Runnable/Callable 设计，继承 Thread 无法融入这套生态。
  • 语法层面：Runnable 作为函数式接口支持 Lambda 表达式，代码更简洁。

继承 Thread 并非全无价值——在极简单的场景、快速验证想法、或者确实需要扩展 Thread 类本身的行为 （例如自定义线程的创建/销毁逻辑）时，它仍然是合法选项。但一旦你的代码从"实验"进入"工程"，就应该切换到 Runnable + 线程池的范式。

一句话总结：把任务（Runnable）和执行的载体（Thread）分开，是 Java 并发编程的第一条设计原则。
