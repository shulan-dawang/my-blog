匿名内部类

2026 年 6 月 8 日

在 Java 中，有些类你只会用一次——比如一个按钮的点击处理器、一个临时的比较规则、或者一个简单的线程任务。 为这些"一次性"的用途去单独定义一个类，像是在为一次性筷子专门建个工厂。匿名内部类（Anonymous Inner Class） 就是为这种场景设计的：在需要的地方就地创建、就地使用，用完即弃。

匿名内部类是什么？

匿名内部类是 Java 中一种特殊的局部内部类，它没有名字，在创建对象的同时完成类的定义。 本质上，它是一个"一次性"的子类或接口实现——定义完立刻实例化，之后再也无法被引用。

基本语法

// 格式：
// new 父类/接口() {
//     // 在这里写类的定义（字段、方法、代码块）
// };

// 一个最简单的例子：实现 Runnable 接口
Runnable task = new Runnable() {
    @Override
    public void run() {
        System.out.println("我是匿名内部类！");
    }
};
task.run();  // 输出：我是匿名内部类！

三行代码就完成了"定义类 + 创建对象"——如果用传统的 named class，你得先定义一个 class MyRunnable implements Runnable，文件多一个，心智负担多一层。

关键特性

  • 没有名字——不能在别处通过类名引用它或创建更多实例。
  • 同时定义和实例化——new 接口() { ... } 一气呵成。
  • 只能继承一个父类或实现一个接口——匿名类不能同时 extends 又 implements。
  • 可以访问外部类的成员——包括外部类的 private 字段和方法。
  • 可以捕获局部变量——但只能捕获 final 或"事实上的 final"变量（Java 8+）。

六大核心使用场景

匿名内部类不是语法糖，它解决的是真实的工程问题。以下六大场景覆盖了日常开发中 90% 的匿名内部类使用：

场景一：事件监听与回调

GUI 编程中最经典的用法——每个按钮、每个菜单项都需要一个事件处理器：

// Swing / Android 中的按钮点击
button.addActionListener(new ActionListener() {
    @Override
    public void actionPerformed(ActionEvent e) {
        System.out.println("按钮被点击了！");
        // 处理点击逻辑...
    }
});

// 文件变更监听
watchService.register(new WatchEvent.Kind[]{StandardWatchEventKinds.ENTRY_MODIFY},
    new WatchHandler() {
        @Override
        public void handle(WatchEvent event) {
            System.out.println("文件发生了变化：" + event.context());
        }
    });

在这种场景下，事件处理器通常只绑定一个控件，用匿名内部类既简洁又能直接访问外部类的成员（比如更新 UI 状态）。

场景二：快速创建线程

当线程逻辑简单且不需要复用时，就地实现 Runnable 或 Callable 是最高效的做法：

// 后台线程执行耗时任务
new Thread(new Runnable() {
    @Override
    public void run() {
        // 执行文件下载
        downloadFile("https://example.com/large-file.zip");
        System.out.println("下载完成！");
    }
}).start();

// 带返回值的 Callable
Future<Integer> future = executor.submit(new Callable<Integer>() {
    @Override
    public Integer call() throws Exception {
        return computeHash("data.txt");
    }
});

场景三：临时排序 / 比较规则

对列表排序时，Comparator 往往是"一次性"的比较规则：

List<String> names = Arrays.asList("张三", "李四", "王五", "赵六");

// 按字符串长度排序
Collections.sort(names, new Comparator<String>() {
    @Override
    public int compare(String s1, String s2) {
        return s1.length() - s2.length();
    }
});

// 按姓名拼音排序
names.sort(new Comparator<String>() {
    @Override
    public int compare(String s1, String s2) {
        return Collator.getInstance(Locale.CHINESE).compare(s1, s2);
    }
});

这种用法在 Java 8 之前是标准做法，现在可以用 Lambda 替代（后面会对比），但理解匿名内部类才能理解 Lambda 的"前身"。

场景四：模板方法中的"钩子"

一些工具类提供了"模板方法"，允许你通过传入实现来定制部分行为：

// 遍历文件夹时，对每个文件执行自定义操作
FileUtils.traverse("C:\\projects", new FileHandler() {
    @Override
    public void onFile(File file) {
        if (file.getName().endsWith(".java")) {
            System.out.println("发现 Java 文件：" + file.getName());
        }
    }

    @Override
    public void onDirectory(File dir) {
        System.out.println("进入目录：" + dir.getName());
    }
});

场景五：单元测试中的 Stub / Mock

写测试时，经常需要快速构造一个接口的"假实现"来隔离被测对象：

// Mock 一个数据源，返回固定数据
UserService userService = new UserService(new UserRepository() {
    @Override
    public User findById(int id) {
        // 不查数据库，直接返回假数据
        return new User(id, "测试用户", "test@example.com");
    }

    @Override
    public List<User> findAll() {
        return Arrays.asList(
            new User(1, "张三", "zhangsan@test.com"),
            new User(2, "李四", "lisi@test.com")
        );
    }
});

// 验证 userService 的行为
User result = userService.getUser(1);
assertEquals("测试用户", result.getName());

场景六：一次性过滤 / 验证规则

// 过滤集合
List<Integer> numbers = Arrays.asList(1, 2, 3, 4, 5, 6);
numbers.removeIf(new Predicate<Integer>() {

// numbers 现在是 [1, 3, 5]

实战核心：用匿名内部类计算方法的执行时间

这是一个在日常开发中非常实用的模式——用一个匿名内部类封装"要执行的代码块"，然后在外部计时。 它体现了匿名内部类的核心设计思想：将行为（要做什么）和基础设施（计时、日志、事务等）分离。

版本一：定义时间计算器

首先，我们设计一个"可计时执行的代码块"的契约：

/**
 * 一个可以被计时执行的代码块。
 * 调用者只需实现 execute()，时间计算由外部框架完成。
 */
interface TimedTask {
    void execute();  // 要执行的业务逻辑
}

然后写一个通用的计时工具：

public class Timer {

    /**
     * 执行任务并计算耗时（秒）。
     *
     * @param task 要执行的代码块（匿名内部类）
     * @return 耗时，单位秒
     */
    public static double run(TimedTask task) {
        long start = System.currentTimeMillis();  // 记录开始时间

        task.execute();  // ← 执行真正的业务逻辑

        long end = System.currentTimeMillis();    // 记录结束时间
        return (end - start) / 1000.0;           // 返回秒数
    }
}

使用时，只需用匿名内部类传入"要执行的代码"：

public class Demo {
    public static void main(String[] args) {

        // ── 测试：计算一段耗时操作的执行时间 ──
        double seconds = Timer.run(new TimedTask() {
            @Override
            public void execute() {
                // ===== 这里写你要测试的代码 =====
                long sum = 0;
                for (int i = 0; i < 100_000_000; i++) {
                    sum += i;
                }
                System.out.println("计算结果：" + sum);
                // =============================
            }
        });

        System.out.printf("执行耗时：%.3f 秒\n", seconds);
    }
}

输出示例：

计算结果：4999999950000000
执行耗时：0.035 秒

这 4 行调用代码就是匿名内部类发挥威力的地方——你要测试的逻辑用匿名类就地写出来，计时逻辑由 Timer.run() 统一处理。 修改要测试的代码时，只改匿名类的内容；要改进计时精度（比如改用 System.nanoTime()），只需改 Timer 一个地方。

版本二：带参数和返回值

上面的版本不能传参也不能返回值，实际业务中经常遇到"输入一个参数，返回一个结果"的场景。 我们用泛型做一个更通用的版本： 

/**
 * 带输入和输出的可计时代码块。
 * @param <T> 输入类型
 * @param <R> 输出类型
 */
interface TimedFunction<T, R> {
    R execute(T input);
}
public class Timer {

    /**
     * 执行并计时（带输入输出）。
     *
     * @param func  要执行的代码块
     * @param input 输入参数
     * @return 包含耗时和执行结果的容器
     */
    public static <T, R> TimedResult<R> run(TimedFunction<T, R> func, T input) {
        long start = System.nanoTime();  // 用 nanoTime 更精确

        R result = func.execute(input);  // 执行业务逻辑

        long end = System.nanoTime();
        double seconds = (end - start) / 1_000_000_000.0;

        return new TimedResult<>(result, seconds);
    }

    // ── 结果容器 ──
    public static class TimedResult<R> {
        public final R result;
        public final double seconds;

        public TimedResult(R result, double seconds) {
            this.result = result;
            this.seconds = seconds;
        }
    }
}

实际使用——比较三种算法的性能：

public class AlgoBenchmark {
    public static void main(String[] args) {
        final int n = 50;

        // ── 测试递归实现 ──
        TimedResult<Long> r1 = Timer.run(
            new TimedFunction<Integer, Long>() {
                @Override
                public Long execute(Integer input) {
                    return fibRecursive(input);
                }
            }, n);
        System.out.printf("递归法  ：fib(%d) = %d，耗时 %.6f 秒\n",
                          n, r1.result, r1.seconds);

        // ── 测试循环实现 ──
        TimedResult<Long> r2 = Timer.run(
            new TimedFunction<Integer, Long>() {
                @Override
                public Long execute(Integer input) {
                    return fibLoop(input);
                }
            }, n);
        System.out.printf("循环法  ：fib(%d) = %d，耗时 %.6f 秒\n",
                          n, r2.result, r2.seconds);

        // ── 测试记忆化递归 ──
        TimedResult<Long> r3 = Timer.run(
            new TimedFunction<Integer, Long>() {
                @Override
                public Long execute(Integer input) {
                    return fibMemo(input, new HashMap<>());
                }
            }, n);
        System.out.printf("记忆法  ：fib(%d) = %d，耗时 %.6f 秒\n",
                          n, r3.result, r3.seconds);
    }

    // 普通递归（O(2^n)）
    static long fibRecursive(int n) {
        if (n <= 1) return n;
        return fibRecursive(n - 1) + fibRecursive(n - 2);
    }

    // 循环迭代（O(n)）
    static long fibLoop(int n) {
        if (n <= 1) return n;
        long a = 0, b = 1;
        for (int i = 2; i <= n; i++) {
            long c = a + b;
            a = b;
            b = c;
        }
        return b;
    }

    // 记忆化递归（O(n)）
    static long fibMemo(int n, Map<Integer, Long> memo) {
        if (n <= 1) return n;
        if (memo.containsKey(n)) return memo.get(n);
        long val = fibMemo(n - 1, memo) + fibMemo(n - 2, memo);
        memo.put(n, val);
        return val;
    }
}

输出对比：

递归法  ：fib(50) = 12586269025，耗时 38.214563 秒
循环法  ：fib(50) = 12586269025，耗时 0.000021 秒
记忆法  ：fib(50) = 12586269025，耗时 0.000412 秒

三种算法一行行对比，计时逻辑完全复用——这就是用匿名内部类封装"要执行的代码块"带来的关注点分离。 算法的代码只管"怎么算"，Timer 只管"花了多久"。

版本三：支持多次运行取平均值

更进一步——同一个任务跑多次，取平均时间，排除 JVM 预热的影响：

public class Timer {

    /**
     * 多次执行并计算平均耗时。
     *
     * @param task  要测试的代码块
     * @param warmupRounds 预热轮数（不计入统计）
     * @param testRounds   正式测试轮数
     * @return 各轮耗时及平均耗时
     */
    public static BenchmarkResult benchmark(TimedTask task,
                                             int warmupRounds,
                                             int testRounds) {
        // ── 预热阶段（不计时） ──
        for (int i = 0; i < warmupRounds; i++) {
            task.execute();
        }

        // ── 正式测量 ──
        double[] times = new double[testRounds];
        double total = 0;
        for (int i = 0; i < testRounds; i++) {
            long start = System.nanoTime();
            task.execute();
            long end = System.nanoTime();
            times[i] = (end - start) / 1_000_000_000.0;
            total += times[i];
        }

        return new BenchmarkResult(times, total / testRounds);
    }

    public static class BenchmarkResult {
        public final double[] eachRound;  // 每轮耗时（秒）
        public final double average;     // 平均耗时（秒）

        public BenchmarkResult(double[] eachRound, double average) {
            this.eachRound = eachRound;
            this.average = average;
        }
    }
}

使用：

BenchmarkResult result = Timer.benchmark(new TimedTask() {
    @Override
    public void execute() {
        // 测试 StringBuilder vs StringBuffer
        StringBuilder sb = new StringBuilder();
        for (int i = 0; i < 100_000; i++) {
            sb.append("hello");
        }
    }
}, 3, 10);  // 预热 3 轮，正式测试 10 轮

System.out.printf("平均耗时：%.6f 秒\n", result.average);
// 输出各轮明细
for (int i = 0; i < result.eachRound.length; i++) {
    System.out.printf("第 %d 轮：%.6f 秒\n", i + 1, result.eachRound[i]);
}

从计时器看匿名内部类的设计思想

这个计时器的例子不止教会了你"怎么计时"——它体现了一个更重要的设计原则：

  1. 定义契约：TimedTask 接口只定义一个 execute() 方法，清晰表达"这是一个可被执行的代码块"。
  2. 匿名实现：每次测试算法时，就地创建一个匿名内部类实现 execute()，把"要测试的代码"写进去。
  3. 外部框架接手：Timer.run() 负责计时、预热、统计——这些横切关注点和业务逻辑完全分离。

这个模式就是模板方法模式（Template Method）的匿名内部类版本——Timer 定义了执行流程（记录开始 → 执行业务 → 记录结束 → 计算耗时）， 其中"业务"这一步由匿名内部类填充。对比传统的模板方法（需要继承），匿名类版本更轻量、更灵活。

匿名内部类 vs Lambda：什么时候用哪个？

Java 8 引入 Lambda 后，很多匿名内部类的场景可以用更简洁的语法替代。但两者并不完全等价：

对比维度 匿名内部类 Lambda 表达式
适用接口 任意接口（多方法也行） 仅函数式接口（只有一个抽象方法）
能否有自己的字段 可以 不可以（只是方法实现）
能否初始化块 可以 不可以
this 指向 匿名内部类实例本身 外部类实例
语法长度 较冗长 极简
编译产物 生成独立的 .class 文件（如 Outer$1.class） 不生成 .class，由 invokedynamic 实现

用 Lambda 重写前面的计时器调用：

// 匿名内部类写法（Java 7 以前）
Timer.run(new TimedTask() {
    @Override
    public void execute() {
        doSomething();
    }
});

// Lambda 写法（Java 8+，前提是 TimedTask 是函数式接口）
Timer.run(() -> doSomething());

一条简单的判断规则：

  • 接口只有一个抽象方法 → 用 Lambda，更简洁。
  • 接口有多个抽象方法 → 用匿名内部类，Lambda 做不到。
  • 需要在实现类中加字段、初始化块 → 用匿名内部类。
  • 需要用到自身的 this（比如把自己注册为监听器） → 用匿名内部类。

使用匿名内部类的注意事项

1. 变量捕获的限制

匿名内部类可以访问外部方法的局部变量，但前提是变量必须是 final 或"事实上的 final"（赋值后不再改变）：

// ✅ 正确 —— message 只赋值了一次
String message = "Hello";
new Thread(new Runnable() {
    @Override
    public void run() {
        System.out.println(message);  // OK
    }
}).start();

// ❌ 错误 —— count 在赋值后又改变了
int count = 0;
count++;  // 这一行让 count 不再是"事实上的 final"
new Thread(new Runnable() {
    @Override
    public void run() {
        System.out.println(count);  // 编译错误！
    }
}).start();

解决方案：用一个数组或可变容器包装变量，绕过变量本身不能变的限制（但不推荐——这破坏了不可变性的意图）。

2. 内存泄漏风险

匿名内部类会隐式持有外部类实例的引用。如果匿名内部类的生命周期比外部类长（比如被提交到线程池长时间运行），外部类就不能被 GC 回收，造成内存泄漏。

public class LeakyClass {

    private byte[] bigData = new byte[100 * 1024 * 1024]; // 100MB

    void startTask() {
        // 这个匿名内部类持有 LeakyClass.this 引用
        executor.submit(new Runnable() {
            @Override
            public void run() {
                // 长时间运行的任务...
                processData(bigData);
            }
        });
    }
    // 即使 LeakyClass 的外部引用没了，bigData 也释放不掉
    // ——因为匿名内部类还抓着外部类的引用
}

如果匿名类不需要访问外部实例成员，可以定义为 静态内部类 或使用 Lambda（Lambda 不持有 this 引用，除非在 Lambda 体内访问了外部实例成员）。

3. 不宜过度使用

如果匿名内部类的代码超过 20 行，或者相同的逻辑要在多处复用，就应该把它提升为独立的命名类。匿名类的设计初衷就是"一次性、短小、就地使用"——太长会严重损害可读性。

小结

今天我们系统学习了匿名内部类的使用场景，并通过"计算方法执行时间"这一实战案例深入理解其设计思想：

  • 匿名内部类是什么：没有名字的局部内部类，在创建对象的同时完成类的定义。"定义 + 实例化"一步到位。
  • 六大场景：事件监听、线程创建、临时比较规则、模板方法钩子、测试 Stub/Mock、一次性过滤规则。
  • 计时实战：通过 TimedTask 接口 + 匿名内部类，将"要执行的代码"和"计时基础设施"彻底分离，体现了模板方法模式的设计思想。
  • 演进路径：从无参无返回 → 带参带返回（泛型） → 多次运行取平均（预热机制），逐步构建一个实用的性能测试小工具。
  • 匿名内部类 vs Lambda：单抽象方法用 Lambda，多方法或有状态需求用匿名内部类。
  • 注意事项：变量捕获的 final 限制、内存泄漏风险、不宜过度使用。

匿名内部类是 Java 中"轻量级模板方法模式"的实现手段。下次当你需要"给一个方法传入一段可执行逻辑"时—— 不是说传一个数据，而是传一段代码——匿名内部类就是为此而生的。
