什么是形式化方法

2026 年 6 月 9 日

如果你问一个经验丰富的架构师"软件开发的本质是什么"，他很可能告诉你两个字：建模。 而形式化方法（Formal Methods）就是建模这件事的"数学版本"——它用严谨的数学语言来描述系统、验证设计、确保正确性。 这篇文章从《大象——Thinking in UML》的视角出发，聊聊形式化思维与软件建模的内在联系。

形式化方法：用数学写"说明书"

形式化方法的核心理念非常简单：用数学语言精确描述软件"应该做什么"，然后用数学推理验证它"确实做到了"。 与之对应的，是我们日常写需求文档、画流程图时使用的"非形式化"（自然语言 + 直觉）方式。

自然语言描述的问题——一个经典例子：

"用户登录时，如果连续输错 3 次密码，账户将被锁定 30 分钟。"

这句话读起来清楚，但藏着大量歧义：

  • "连续"——是不是只要中间有一次成功就重置计数器？
  • "3 次"——有没有时间窗口？还是一辈子累计？
  • "锁定"——用户收到什么提示？管理员能否提前解锁？
  • "30 分钟"——锁定期间再尝试会重置倒计时吗？

形式化方法要解决的就是这个问题——用无歧义的数学语言写需求，让"我以为你懂了"变成"我们精确地达成一致"。

形式化方法的三大流派

1. 形式化规约（Formal Specification）

用数学语言写"这个系统应该做什么"。最著名的语言包括 Z 语言、VDM、B 方法。它们用集合论和一阶谓词逻辑来描述系统的状态和行为： 

// 一个用数学风格描述的"栈"的规约
// （不是真正的 Z 语言，但传达了其精神）

状态定义：
  Stack ::= seq of Item          // 栈就是一个序列
  maxSize : ℕ                    // 最大容量

操作——Push：
  前置条件：length(stack) < maxSize
  后置条件：stack' = stack ⌢ [item]   // 新栈 = 旧栈末尾追加元素

操作——Pop：
  前置条件：length(stack) > 0
  后置条件：stack = stack' ⌢ [result]  // 旧栈 = 新栈末尾追加结果

这种描述没有任何歧义——"前置条件"精确说明了操作何时合法，"后置条件"精确说明了操作的效果。 数学家、开发者、测试人员都能从中得到同一套理解。

2. 模型检验（Model Checking）

把系统抽象成一个有限状态模型（状态 + 状态之间的转移），然后用工具穷举所有可能的状态序列， 检查是否违反了给定的性质（如"永远不会死锁""缓冲区永远不会溢出"）。

核心流程：

  1. 建模：将系统抽象为状态机。
  2. 规约：用时序逻辑（如 LTL、CTL）表达期望的性质——例如"最终一定会收到响应"。
  3. 验证：模型检验器自动穷举所有状态路径，检查性质是否恒成立。
  4. 反例：如果性质不成立，工具给出一个具体的反例路径——"你看，从状态 A 走到状态 B 再走到状态 C 时，死锁了"。

模型检验在硬件验证（Intel/AMD 的芯片设计）、通信协议验证（TCP/IP 的安全性质）、 以及航空航天软件中有着广泛应用——因为在这些领域，bug 的代价是数十亿美元甚至人命。

3. 定理证明（Theorem Proving）

把程序和规约都表示为数学公式，然后像做数学证明题一样，一步步推导证明"程序满足规约"。 与模型检验不同，定理证明不受状态爆炸的限制，但需要大量的专业知识和人工引导。

知名的定理证明工具：Coq、Isabelle/HOL、Agda。近年来，Rust 社区也出现了基于定理证明的验证工具（如 Verus）。

形式化方法离我们有多远？

大多数 Java 程序员一辈子没碰过形式化方法——这是事实。但这不意味着它无关紧要。实际上，它的影响力是"渗透式"的：

  • 类型系统本身就是一种形式化方法——Java 的泛型、Kotlin 的可空类型、Rust 的所有权系统，本质上都是类型论（Type Theory）的形式化应用。类型检查就是最普及的"轻量级形式化验证"。
  • 设计模式的形式化——GoF 的 23 种设计模式，最初是用自然语言描述的。后来研究者用形式化语言给出了精确的定义——模式不是"经验之谈"，而是有严格数学性质的结构。
  • Amazon/AWS 的部分核心服务使用 TLA+（一种形式化规约语言）来设计——S3、DynamoDB 的分布式一致性算法在写代码前先用 TLA+ 建模验证过。
  • 智能合约审计——以太坊上的 DeFi 协议越来越多地使用形式化验证来保证资金安全。

UML：从形式化到可视化

如果说形式化方法是"用数学为软件建立精确模型"，那么 UML（统一建模语言）就是把建模这件事从数学家手里拿过来，交给所有开发者。

UML 的创造者之一 Grady Booch 曾说过：形式化方法和 UML 不是对立的，而是同一个问题在不同抽象层次上的解决方案。 形式化方法关心"精确到什么程度"，UML 关心"理解到什么程度"—— 一个偏向数学家的严谨，一个偏向开发者的直观。

UML 的"半形式化"特性

UML 不是完全形式化的——它的语法有精确的定义（你可以用 XMI 格式序列化 UML 模型）， 但很多语义部分是开放的（比如"用例"的粒度、"关联"的导航方向）。 这正是 UML 的设计哲学：在需要精确的地方精确（类图、状态图），在需要灵活的地方灵活（用例图、部署图）。

UML 的图中，形式化程度从高到低大致排列：

  1. 对象图 / 通信图——直接反映运行时的对象关系，几乎可以机械地映射为代码。
  2. 类图 / 包图——结构清晰，有严格语法，可以正向/逆向工程。
  3. 状态图 / 活动图——本质是状态机和流程图，可以用形式化方法验证。
  4. 顺序图 / 时序图——表达交互，有一定结构但细节灵活。
  5. 用例图 / 部署图——主要是沟通工具，形式化最弱。

《大象——Thinking in UML》：建模不是画图，是思考

谭云杰的《大象——Thinking in UML》是一本在中国软件圈里影响深远的 UML 经典。它的书名本身就是核心观点—— 大象（无形），Thinking（思维），UML 的本质不是学会画那些框框线线，而是学会用建模的方式思考问题。

核心思想一：建模是"翻译"，不是"记录"

书中反复强调一个观点：建模不是把需求文档抄成 UML 图——那只是换了一种格式。真正的建模是把现实世界中的业务逻辑"翻译"成软件世界的抽象结构。 这个翻译过程需要理解、裁剪、抽象、重构，就像一位翻译家在两种语言间寻找最精准的表达。

这和形式化方法的精神完全一致——形式化方法的核心就是把"模糊的需求"翻译成"精确的数学表述"。UML 则是在"精确"和"可理解"之间寻找平衡点。

核心思想二：用例驱动——一切从"谁要做什么"开始

书中花了大量篇幅讲用例（Use Case）。作者认为，用例不是功能列表，而是从用户视角重新定义系统的价值边界。 很多项目失败不是因为代码写得差，而是因为从一开始就没搞清楚"到底为谁解决了什么问题"。

用例建模的核心步骤（取自书中）：

  1. 找出参与者（Actor）——谁在使用系统？人？外部系统？定时任务？
  2. 找出用例——参与者想达到什么目的？（不是"系统要有什么功能"，而是"用户要完成什么事"）
  3. 建立用例之间的关系——包含（include）、扩展（extend）、泛化。
  4. 用活动图/顺序图精细化用例场景——把一句话的用例展开为完整的业务流。

书中一个经典例子：

一个银行系统的"转账"不是一个用例——"转账"是银行柜员视角的词汇。真正的用例是参与者 × 目的的组合：
"储户——从我的账户向另一账户转出资金"
"银行柜员——代客户办理跨行转账"
"系统管理员——撤销一笔可疑转账"

同一个业务操作，不同的参与者有不同的关注点——这就是用例建模的深度。

核心思想三：从分析到设计——建立"对象世界"

书中将建模过程分为几个递进的层次：

阶段 核心问题 产物
业务建模 现实世界怎么运作的？ 业务用例、业务对象模型
系统用例 系统要帮谁做什么？ 系统用例图、用例规约
分析模型 系统内部需要哪些"概念类"？ 分析类图、包图
设计模型 这些类在技术上怎么实现？ 设计类图、顺序图、状态图
实现 代码怎么写？ Java / C# / Python 代码

这个分层思想极其重要——每一层只关心自己层次的问题。业务建模时不纠结技术细节， 设计建模时不纠结业务概念——层层递进，每一步的输入都是上一步的输出。

核心思想四：抽象能力——建模的灵魂

书中反复强调，建模工具可以一周学会，但建模能力需要长年积累。建模的本质是抽象：

  • 分类——把相似的东西归为一类，定义一个"概念"（如"账户"不是某一个具体的账户，而是所有账户的共同结构）。
  • 泛化——从多个类中提取共同特征，形成父类（如"活期账户"和"定期账户"泛化为"账户"）。
  • 关联——发现事物之间的联系（如"客户拥有多个账户"）。
  • 边界——决定什么在系统之内、什么在系统之外。

这个抽象过程，和形式化方法中"从具体实现中提炼数学规约"的精神如出一辙。

形式化方法与 UML：互补而非对立

形式化方法和 UML 建模解决的是软件开发的同一个根本问题——如何确保我们构建的系统是正确的—— 但走了不同的路径：

对比维度 形式化方法 UML 建模（大象思维）
语言 数学（集合论、谓词逻辑、时序逻辑） 图形 + 自然语言
精确度 完全精确，无歧义 半形式化，有限制的灵活性
可验证性 可机器验证（模型检验、定理证明） 可人工评审，部分可自动检查
沟通门槛 高（需要数学背景） 中等（需要建模训练，但不需数学基础）
适用场景 安全关键系统（航空、医疗、芯片） 企业应用、业务系统、大多数商业软件
核心价值 "我们可以证明这个系统没有某类 bug" "我们所有人对系统有统一的理解"

在现实中，两者常常结合使用。一个成熟的软件工程实践可能是：用UML建模来沟通和达成共识， 然后在关键模块（如支付状态机、分布式一致性协议）上使用形式化方法进行精确验证。

一个具体例子：从 UML 状态图到形式化验证

以订单生命周期为例——先用 UML 状态图描述，再转化为形式化表示：

// ── UML 状态图（文字描述） ──
// 待支付 → [支付成功] → 已支付
// 待支付 → [超时 30min] → 已取消
// 已支付 → [发货] → 已发货
// 已发货 → [确认收货] → 已完成
// 待支付 → [用户取消] → 已取消

// ── 转化为形式化状态定义 ──
type OrderState = PENDING | PAID | SHIPPED | COMPLETED | CANCELLED

// ── 形式化转移函数 ──
transitions(PENDING)  = { pay → PAID,
                          timeout → CANCELLED,
                          cancel → CANCELLED }
transitions(PAID)     = { ship → SHIPPED }
transitions(SHIPPED)  = { confirm → COMPLETED }
transitions(COMPLETED) = {}
transitions(CANCELLED) = {}

// ── 可以形式化验证的性质 ──
// 1. 可达性：从 PENDING 出发，COMPLETED 是否可达？→ 是
// 2. 安全性：是否存在从 COMPLETED 回到 PENDING 的路径？→ 否 ✓
// 3. 活性：PENDING 状态下，pay 操作是否始终可触发？→ 是 ✓
// 4. 死锁检查：是否存在某个状态没有任何可用转移？→ COMPLETED 和 CANCELLED 是终态，合理

这就是 UML 和形式化方法配合的经典模式：UML 负责沟通和理解，形式化负责验证和保证。

对 Java 学习者的启示

你可能会问：我是一个 Java 学习者，形式化方法和我有什么关系？答案是：关系很大。

1. 类型系统是日常的形式化

每次你写 List<String> 而不是 List（裸类型），你就在用类型论的形式化思想来约束代码—— "这个列表只能放字符串"就是一个形式化的断言，编译器帮你验证它。

2. 契约式编程（Design by Contract）

Java 的 assert、Guava 的 Preconditions、Spring 的 @NotNull——这些都是形式化方法中 "前置条件 / 后置条件 / 不变式"思想在工程中的落地。 

// 这本质上就是"形式化规约"的 Java 版：
public void transfer(Account from, Account to, BigDecimal amount) {
    // 前置条件
    Objects.requireNonNull(from, "转出账户不能为空");
    Objects.requireNonNull(to, "转入账户不能为空");
    if (amount.compareTo(BigDecimal.ZERO) <= 0) {
        throw new IllegalArgumentException("金额必须大于 0");
    }
    if (from.getBalance().compareTo(amount) < 0) {
        throw new InsufficientBalanceException("余额不足");
    }

    // 执行转账...
    from.debit(amount);
    to.credit(amount);

    // 后置条件（用 assert 验证）
    assert from.getBalance().compareTo(oldFromBalance.subtract(amount)) == 0;
    assert to.getBalance().compareTo(oldToBalance.add(amount)) == 0;
}

3. 单元测试是"穷举检验"的简化版

模型检验穷举所有状态——这对大多数业务系统来说不可能（状态空间太大）。但单元测试做的是同样的事： 定义输入、预期输出、边界条件，系统性地验证行为。好的测试用例设计，本质上就是在做一个"微型的形式化验证"。

4. 学会建模，才能写好 Java

《大象——Thinking in UML》传递的最重要的信息是：代码只是最后一步，思维才是整个过程。 一个 Java 程序员如果只会写语法，就是在搬砖；如果他能在脑子里完成"业务→用例→分析模型→设计模型→代码"这一整套思维过程， 他就有了架构师的素养。形式化方法和 UML 建模，都是训练这种思维方式的工具。

小结

这篇文章从形式化方法的概念出发，结合《大象——Thinking in UML》的核心思想，梳理了精确建模与可视化建模之间的关系：

  • 形式化方法是用数学语言精确描述和验证软件——三大流派：形式化规约、模型检验、定理证明。用在安全关键系统（航空、芯片、分布式协议），但思想渗透在日常开发中（类型系统、契约式编程、单元测试）。
  • UML 是"半形式化"的——在精确性和可理解性之间取平衡。它让建模这件事走出数学家的书房，进入所有开发者的日常。
  • 《大象——Thinking in UML》的核心观点：建模不是画图，是思考。从用例驱动出发，经过业务建模→系统用例→分析模型→设计模型→实现的层层递进，最终建立对系统的精确理解。
  • 二者互补：UML 负责沟通和理解，形式化方法负责验证和保证。一个成熟的项目在关键模块上结合使用两者。
  • 对 Java 学习者而言：泛型类型检查、契约式编程、单元测试——这些都是形式化思想在日常编码中的影子。"学会建模，才能写好代码"。

最后引用《大象》书中的一句话作结："软件世界的构建，始于你对现实世界的理解。" 形式化方法给你理解世界的精确语言，UML 给你理解世界的可视化工具——两者结合，才是完整的建模能力。
