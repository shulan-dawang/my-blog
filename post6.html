文件复制：字符缓冲流与字节缓冲流

2026 年 6 月 19 日

文件复制是日常开发中最常见的 I/O 操作之一。Java 提供了丰富的 I/O 流类库，其中缓冲流是提升读写效率的关键手段。 这篇文章从实战出发，手把手用字符缓冲流和字节缓冲流实现文件复制，然后深入对比两者的差异和适用场景。

为什么要用缓冲流？

在理解缓冲流之前，先看一个关键事实：每次磁盘 I/O 操作都非常昂贵。如果你每次只读一个字节就交给程序处理， 复制一个 10MB 的文件就要发起 1000 多万次磁盘读写——这就像每次只搬一块砖，效率极低。

缓冲流的解决思路就像用小推车搬砖：在内存中开辟一块缓冲区（默认 8192 字节），一次从磁盘拉一车数据到内存， 程序从内存中取数据；写数据时也是先攒满一车，再一次性写到磁盘。这样磁盘 I/O 的次数就从"每次一个字节"变成了"每 8192 字节一次"—— 效率提升了数千倍。

方案一：字节缓冲流（BufferedInputStream / BufferedOutputStream）

字节流以字节（byte）为最小读写单位，可以处理任意类型的文件——文本、图片、视频、音频、PDF，通通不在话下。

核心类：BufferedInputStream 和 BufferedOutputStream，它们分别包装 FileInputStream 和 FileOutputStream。

import java.io.*;

public class BufferedByteCopy {
    public static void main(String[] args) {
        String source = "source.jpg";    // 源文件
        String target = "target.jpg";    // 目标文件

        try (BufferedInputStream bis = new BufferedInputStream(
                 new FileInputStream(source));
             BufferedOutputStream bos = new BufferedOutputStream(
                 new FileOutputStream(target))) {

            byte[] buffer = new byte[8192];   // 8KB 缓冲区
            int bytesRead;

            while ((bytesRead = bis.read(buffer)) != -1) {
                bos.write(buffer, 0, bytesRead);  // 写多少读多少
            }

            System.out.println("文件复制完成！");

        } catch (IOException e) {
            e.printStackTrace();
        }
    }
}

代码拆解：

  • try-with-resources：自动关闭流，不用手动 finally { close(); }。
  • BufferedInputStream 包装 FileInputStream：把原始的字节读取加上缓冲区。
  • BufferedOutputStream 包装 FileOutputStream：写数据时先写入内部缓冲区，满了再一次性刷到磁盘。
  • 自定义 8KB 数组：外层再套一个字节数组作为传输容器，进一步减少 read() 调用次数。
  • bos.write(buffer, 0, bytesRead)：注意用 bytesRead 而不是 buffer.length——最后一次读取可能填不满整个数组。

方案二：字符缓冲流（BufferedReader / BufferedWriter）

字符流以字符（char）为最小读写单位，专用于文本文件（.txt、.java、.html、.json 等）。 它内部会自动处理字符编码（如 UTF-8、GBK），把字节和字符之间的转换对开发者透明。

核心类：BufferedReader 和 BufferedWriter，分别包装 FileReader 和 FileWriter。

import java.io.*;

public class BufferedCharCopy {
    public static void main(String[] args) {
        String source = "source.txt";    // 源文本文件
        String target = "target.txt";    // 目标文本文件

        try (BufferedReader br = new BufferedReader(
                 new FileReader(source));
             BufferedWriter bw = new BufferedWriter(
                 new FileWriter(target))) {

            char[] buffer = new char[8192];   // 8KB 字符缓冲区
            int charsRead;

            while ((charsRead = br.read(buffer)) != -1) {
                bw.write(buffer, 0, charsRead);
            }

            System.out.println("文本文件复制完成！");

        } catch (IOException e) {
            e.printStackTrace();
        }
    }
}

代码结构和字节缓冲流几乎一样，区别在于：

  • 数据单位：char[] 替代 byte[]，每个元素是 2 字节的 Unicode 字符。
  • 底层流：FileReader / FileWriter 替代 FileInputStream / FileOutputStream。
  • 编码处理：FileReader 使用系统默认编码；如需指定编码，用 InputStreamReader 替代。

BufferedReader 的独门利器：readLine()

BufferedReader 有一个字节流没有的实用方法——readLine()，一次读取一整行文本，返回 String。 这让按行处理文本变得极简： 

// 按行复制（利用 readLine）
try (BufferedReader br = new BufferedReader(new FileReader("source.txt"));
     BufferedWriter bw = new BufferedWriter(new FileWriter("target.txt"))) {

    String line;
    while ((line = br.readLine()) != null) {
        bw.write(line);
        bw.newLine();   // 写入换行符（跨平台）
    }
}

注意 readLine() 返回的字符串不包含换行符，所以写入时需要用 newLine() 补上。 newLine() 会根据平台自动选择 \n（Linux/macOS）或 \r\n（Windows），比手动拼接换行符更稳妥。

字符缓冲流与字节缓冲流：深度对比

表面上看，两者都是"加一层缓冲区提升 I/O 效率"，但底层机制和适用场景有本质区别。 下面从多个维度深入对比：

1. 数据单位

对比维度 字节缓冲流 字符缓冲流
最小单位 byte（8 位） char（16 位 Unicode）
缓冲区类型 byte[] char[]
一次读取 N 个字节 N 个字符

2. 适用文件类型

字节流是"万能"的——任何文件归根到底都是字节序列。图片的每个像素、音频的每个采样点、视频的每一帧，最终存储形式都是二进制字节。字节流原样搬运，不做任何转换，保证数据完整性。

字符流是"文本专用"的——它在读写过程中会自动进行编码转换。读文本时把字节按编码（如 UTF-8）解码为字符，写文本时把字符按编码转为字节。 这个"隐式转换"是利器也是陷阱：处理文本时省心，处理二进制文件时会破坏数据。

一个血泪教训：用字符流复制图片：

// 错误示范 —— 千万别这么做！
FileReader fr = new FileReader("photo.jpg");   // 把图片字节当成字符解码
FileWriter fw = new FileWriter("copy.jpg");
// 结果：复制出来的图片打不开，文件损坏

原因：图片中的某些字节序列在 UTF-8 解码时会被映射为"替换字符"（U+FFFD），或者触发编码异常。 字节被"翻译"了，不复原样——这就是字符流不能碰二进制文件的原因。

3. 编码处理

场景 字节缓冲流 字符缓冲流
编码感知 完全不关心编码，原样搬运字节 内置编码转换，默认系统编码
指定编码 不需要 new InputStreamReader(fis, "UTF-8")
跨平台风险 无 默认编码在不同 OS 可能不同

指定编码的正确姿势（避免乱码）：

// 用 UTF-8 读，用 UTF-8 写 —— 编码保持一致
try (BufferedReader br = new BufferedReader(
         new InputStreamReader(new FileInputStream("source.txt"), "UTF-8"));
     BufferedWriter bw = new BufferedWriter(
         new OutputStreamWriter(new FileOutputStream("target.txt"), "UTF-8"))) {

    String line;
    while ((line = br.readLine()) != null) {
        bw.write(line);
        bw.newLine();
    }
}

4. 特有功能

字符缓冲流有一个字节流不具备的"杀手锏"——readLine()：

  • 按行读取：一次返回一行 String，天然适合处理日志、CSV、配置文件等行式文本。
  • 按行写入：配合 newLine()，自动处理跨平台换行符。

字节流要想实现"读一行"，需要自己维护一个 StringBuilder，逐字节判断 \n，麻烦且易出错。

5. 性能对比

在处理文本文件时，两者的效率差异通常可以忽略——因为真正的瓶颈在磁盘 I/O，不在内存中的字节/字符转换。 但在极大规模的文本处理场景下，字节流略快一点（少了一层编码转换），代价是编码处理要自己来。

一个简单的性能测试（复制 50MB 的文本文件，单位：毫秒）：

方法 耗时（约）
无缓冲的 FileInputStream/FileOutputStream 12000+ ms
BufferedInputStream / BufferedOutputStream ~400 ms
BufferedReader / BufferedWriter ~450 ms

关键结论：缓冲才是性能提升的绝对主力（快了 30 倍），字节流 vs 字符流之间的差异只有约 10%。 所以选型时不必纠结性能，按文件类型选就好——文本用字符流，二进制用字节流。

6. 继承体系一览

字节流体系：                            字符流体系：
InputStream  (抽象)                      Reader  (抽象)
  ├─ FileInputStream                       ├─ InputStreamReader
  │    └─ BufferedInputStream 装饰          │    └─ FileReader
  └─ ByteArrayInputStream                  ├─ BufferedReader 装饰
                                           └─ CharArrayReader
OutputStream (抽象)
  ├─ FileOutputStream                   Writer  (抽象)
  │    └─ BufferedOutputStream 装饰         ├─ OutputStreamWriter
  └─ ByteArrayOutputStream                 │    └─ FileWriter
                                           ├─ BufferedWriter 装饰
                                           └─ CharArrayWriter

注意这个经典的装饰器模式：BufferedInputStream 不是继承 FileInputStream， 而是包装任意一个 InputStream——这正是"组合优于继承"的教科书级应用。你可以灵活组合： new BufferedInputStream(new FileInputStream("a.bin"))。

如何选择：一张图就够了

简化成一条判断链：

  1. 文件是纯文本吗？（.txt / .java / .json / .xml / .html / .csv …）
    是 → 用 BufferedReader + BufferedWriter（需要指定编码时用 InputStreamReader/OutputStreamWriter 包装）。
    否 → 进入第 2 步。
  2. 文件是二进制吗？（图片 / 视频 / 音频 / PDF / .class / .jar …）
    是 → 用 BufferedInputStream + BufferedOutputStream。
    不确定 → 也请用字节流，至少不会损坏数据。

完整实战：通用文件复制工具类

最后，把知识整合成一个实用的工具方法——自动判断文件类型，选择合适的流：

import java.io.*;

public class FileCopyUtil {

    /**
     * 复制文件。文本文件用字符缓冲流，其他用字节缓冲流。
     *
     * @param source 源文件路径
     * @param target 目标文件路径
     */
    public static void copy(String source, String target) {
        if (isTextFile(source)) {
            copyText(source, target);
        } else {
            copyBinary(source, target);
        }
    }

    // ── 字节缓冲流复制（通用） ──
    private static void copyBinary(String source, String target) {
        try (BufferedInputStream in = new BufferedInputStream(
                 new FileInputStream(source));
             BufferedOutputStream out = new BufferedOutputStream(
                 new FileOutputStream(target))) {

            byte[] buf = new byte[8192];
            int len;
            while ((len = in.read(buf)) != -1) {
                out.write(buf, 0, len);
            }
            System.out.println("字节流复制完成 → " + target);

        } catch (IOException e) {
            e.printStackTrace();
        }
    }

    // ── 字符缓冲流复制（文本专用） ──
    private static void copyText(String source, String target) {
        try (BufferedReader in = new BufferedReader(
                 new InputStreamReader(
                     new FileInputStream(source), "UTF-8"));
             BufferedWriter out = new BufferedWriter(
                 new OutputStreamWriter(
                     new FileOutputStream(target), "UTF-8"))) {

            String line;
            while ((line = in.readLine()) != null) {
                out.write(line);
                out.newLine();
            }
            System.out.println("字符流复制完成 → " + target);

        } catch (IOException e) {
            e.printStackTrace();
        }
    }

    // ── 简单判断是否是文本文件 ──
    private static boolean isTextFile(String path) {
        String lower = path.toLowerCase();
        return lower.endsWith(".txt")  || lower.endsWith(".java") ||
               lower.endsWith(".json") || lower.endsWith(".xml")  ||
               lower.endsWith(".html") || lower.endsWith(".css")  ||
               lower.endsWith(".js")   || lower.endsWith(".csv")  ||
               lower.endsWith(".md")   || lower.endsWith(".yaml") ||
               lower.endsWith(".py")   || lower.endsWith(".sql");
    }

    // ── 测试 ──
    public static void main(String[] args) {
        copy("note.txt",   "note_backup.txt");    // → 字符流
        copy("photo.jpg",  "photo_backup.jpg");   // → 字节流
        copy("report.pdf", "report_backup.pdf");  // → 字节流
    }
}

这个工具类体现了本文的核心思想：根据文件类型选择合适的流，字符流处理文本简洁优雅，字节流处理二进制安全可靠。

小结

今天我们从文件复制这个实际需求出发，系统梳理了缓冲流的用法和选型：

  • 为什么要缓冲：磁盘 I/O 昂贵，缓冲把"每次搬一块砖"变成"用小推车搬一车砖"，效率提升数十倍。
  • 字节缓冲流：BufferedInputStream / BufferedOutputStream，万能通用，适合所有文件类型，原样搬运不转码。
  • 字符缓冲流：BufferedReader / BufferedWriter，文本专用，内置编码处理，readLine() 按行读取极方便。
  • 核心区别：字符流在读写过程中做编码转换（字节 ↔ 字符），适合文本；字节流不做转换，适合二进制。字符流读图片会损坏数据。
  • 选型口诀：文本用字符流，其他全用字节流。

这两个流体系是 Java I/O 的基石，理解它们的差异和协作方式，后续学习网络流、NIO、序列化都会轻松很多。 最后记住：缓冲是性能的灵魂，选对类型是正确性的保证——性能再快，数据错了也白搭。
